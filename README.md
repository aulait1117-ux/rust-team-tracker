# Rustゲームチーム管理ツール

Rust（サバイバルゲーム）のコミュニティサーバー「Rustoria.co US Main」で遊ぶチーム向けの、敵チームのオンライン状況監視・チームメンバーの資源ファーム量ランキングツール。

**ゲームクライアント自体を自動操作する機能は一切ない**。BattleMetricsの公開APIと、rustoria.co自身が一般公開しているリーダーボードデータを読み取り、見やすく整理して表示するだけの監視・記録ツール。

- 元々は社長個人ツール（`99_荒川プライベート/`）として開発、2026-07-06に企業道開発部の正式業務に移管
- リポジトリ：`github.com/aulait1117-ux/rust-team-tracker`（本ファイルが置かれている場所とは別のGitHub Pagesリポジトリ。企業道メインリポジトリの`.gitignore`に登録済み）

## 公開URL

| ページ | URL | 内容 |
|---|---|---|
| 敵チーム監視 | https://aulait1117-ux.github.io/rust-team-tracker/index.html | 味方・敵チームのオンライン状況をリアルタイム監視 |
| ファームランキング | https://aulait1117-ux.github.io/rust-team-tracker/ranking.html | チームメンバーの資源ファーム量・プレイ時間ランキング |

両ページとも同じロック画面パスワードと同じルームコードを共有する（片方でルームに参加すれば、もう片方も自動的に同じルームに繋がる）。

## 初回アクセス

1. 上記URLのどちらかを開く
2. まずルームコード入力画面が出る。既存チームのルームコードを持っている人から教えてもらって「参加する」を押すか、初めての場合は「🆕 新しいルームを作る」の前にこのルーム専用のパスワードを決めて入力する
3. 参加する場合は、コード入力後にそのルーム専用のパスワードを聞かれる（ルームごとに別のパスワード。本物の暗号学的セキュリティではなく簡易的な目隠しだが、ルームごとに分かれているので1件漏れても他のルームには影響しない）
4. 一度パスワードを通すと、ブラウザに記憶されるので次回以降は自動的に同じルームへ繋がる
5. 友達を招待するときは「📋 コピー」でルームコードをコピーし、パスワードは別途口頭等で伝える
6. パスワードを変更したい場合は「🔑 パスワード変更」から新しいパスワードに更新できる（そのルームの全メンバーに新しいパスワードを伝え直す必要がある）

> 2026-07-06以前に作られた既存ルーム（社長個人チームの`3GL5YX`など）は、まだ専用パスワードが設定されていないため、引き続き旧共通パスワード `0611` で入れる。新しいルームは作成時に決めた専用パスワードのみが有効。

## 🛡 敵チーム監視（index.html）の使い方

### できること
- 登録した名前（味方・敵どちらも）が監視中のサーバーにログインしているかを、BattleMetrics公開APIで45秒おきに自動確認
- オンラインになった瞬間に通知音・デスクトップ通知
- 各メンバーの「ログイン傾向」（何時にログインしやすいか）を過去の記録からグラフ表示
- スクリーンショットからOCRで名前を一括抽出して登録（対応言語：英語・日本語）

### 監視サーバーの変更
ヘッダーの「🔄 サーバーを変更」を押すと、BattleMetrics上の任意のRustサーバーを名前で検索して切り替えられる（Rustoria.co運営の26サーバーに限らず、BattleMetricsに載っている全Rustサーバーが対象）。既定では「Rustoria.co」で検索した結果が表示される。選択はブラウザに記憶される。ranking.html側のサーバー選択とは別の仕組み（BattleMetricsとrustoria.coはサーバーIDの体系が別なので、双方で同じサーバーを選ぶ場合は手動で名前を合わせる必要がある）。

### 基本操作
1. 「＋ チーム追加」でチーム（グループ）を作る。種別は「🟢 味方」か「⚠ 敵」を選ぶ
2. チームカード内の「メンバー追加」欄に名前を入力して追加。BattleMetrics上の候補が出てくるので、正しい人を選ぶと**BattleMetrics IDとの紐付け**が行われる（紐付けると、後で改名されても自動で追従する）
3. 候補が出ない・とりあえず名前だけ登録したい場合は「候補なし（そのまま追加できます）」でも追加できる（後から🔗ボタンでID紐付けし直せる）
4. 各メンバーの行に🟢 IN / ⚫ OFF のオンライン状況が自動表示される
5. Steam ID欄（任意）：BattleMetrics連携とは別に、参照用としてSteam IDかプロフィールURLを入力できる。入力すると🔗マークが出てSteamプロフィールを開ける（同名の別人を見分けたい時に使う）

### Steam IDでBattleMetrics IDを直接紐付けられない理由
BattleMetricsの公開検索APIは名前でしか検索できず、Steam ID・プロフィールURLでは検索できない（実際にテスト済み）。そのため名前検索での紐付けのみ対応している。

### バックアップ
ルームバーの「📥 バックアップをダウンロード」で、現在のチーム・メンバー一覧をJSONファイルとしてダウンロードできる。Firebaseが万一使えなくなった場合の保険。

## 📊 ファームランキング（ranking.html）の使い方

### 表示・記録する項目
資源8種類（**木・石・金属鉱石・硫黄鉱石・上金（HQM）・皮・脂肪・骨**）＋キル・デス・プレイ時間。rustoria.co自身の統計API（`/statistics/leaderboards/{server}/resources`）がこの8項目をまとめて返す仕様になっているため、これに合わせている。キル・デスもrustoria.co側の別バンドル（`pvp`統計、下記参照）から取得できる。

### サーバー選択
画面上部の「サーバー」プルダウンで、rustoria.coが運営する約26サーバー（US Main・EU Main・SEA Main・各種倍率サーバー等）から選べる。既定はVanilla Main US。選択はブラウザに記憶される。サーバーを切り替えると、そのサーバーのワイプ履歴・資源データを取得するようになる。

### シーズン（ワイプ）選択
「サーバー」の右にある「シーズン（ワイプ）」プルダウンで、選んだサーバー側の実際のワイプ履歴（週1回、日付一覧）から選べる。

- 選んだワイプの記録だけが表に表示される（他のワイプの記録は隠れるだけで消えない）
- 新規追加した記録は、その時点で選んでいるサーバー・ワイプに自動的に紐付く
- 同じ人でもワイプが変われば新しい行として記録される（資源はワイプごとにリセットされるゲーム内の仕様に合わせている）。ワイプの日付はサーバーをまたいで一意ではない（同じ日に複数サーバーがワイプすることがある）ため、同名・同Steam IDによる「上書き」判定は選んでいるサーバー・ワイプの組み合わせの中でだけ行われる

### 記録する3つの方法

**① 🌾 自動取得（推奨）**
名前欄に入力して「🌾 自動取得」を押すと、rustoria.co側から名前検索して**資源8種・キル・デスをまとめて**自動取得する（`resources`統計バンドルと`pvp`統計バンドルを同時に取得）。

- 中継サーバー（Cloudflare Worker）経由で取得（rustoria.co側がブラウザからの直接アクセスを許可していないため）
- 今選んでいるサーバー・ワイプのデータを取得する（過去のワイプを選んでいれば、そのワイプ時点の記録が取れる）
- 同名候補が複数見つかった場合は自動入力せず「同名候補がX件見つかった」と警告する（別人のデータを誤って入れないための安全装置）。資源・キルデスそれぞれ独立に判定するため、片方だけ見つかって片方は見つからない、ということもある
- 連続クリック防止のため、1回押すと3秒間は再度押せない

**② 📋 コピペして自動入力**
rustoria.co/leaderboardsの該当ページを開き、資源の表示部分をコピー（Ctrl+C）して貼り付け欄にペースト→「🔍 解析してフォームに入力」を押すと、8項目のうち見つかった数値だけ自動で読み取ってフォームに入れる。ネット通信は一切発生しない（純粋にブラウザ内でのテキスト解析）。**資源のみ対応**（キル・デスを自動で取得したい場合は🌾自動取得ボタンを使う）。

**③ 手入力**
どれも使わず、数値を直接入力してもよい。

### プレイ時間の自動取得
名前を入力して「⏱ 自動取得」を押すと、まずBattleMetrics本家の「プレイ時間」ランキング（全件、認証不要の公開API）を検索する。見つからない場合は、敵チーム監視ツールがローカルに記録しているオンライン時間の合計（セッション記録の合計）にフォールバックする。どちらにも記録がない場合は「見つからなかった」と表示され、手入力が必要。

ローカル記録は敵チーム監視ツールが動いている間しか記録されないため、45秒間隔の誤差や、ツールを起動していない間のオフライン検知漏れによる誤差がある（BattleMetrics本家の記録より精度は落ちる）。

### データ取得元ラベル
テーブルの「合計ファーム」「プレイ時間」の横に小さいアイコンが付く。どこから取得した数値かを表す：

| アイコン | 意味 |
|---|---|
| 🌾 | rustoria.coから自動取得 |
| 📋 | コピペ解析 |
| ⏱ | BattleMetricsから自動取得（プレイ時間） |
| 🧮 | ローカル記録からの推定（プレイ時間） |
| （なし） | 手入力 |

アイコンにカーソルを合わせるとツールチップで詳しい取得元が出る。

### 更新履歴
「🕘 更新履歴を見る」を押すと、誰がいつ何を追加・更新・削除したかの記録（最大200件）が見られる。

### 削除
🗑ボタンを押すと確認ダイアログが出る。「OK」を押さないと削除されない。

### バックアップ
「📥 バックアップをダウンロード」で、現在の全ランキングデータ＋更新履歴をJSONファイルとしてダウンロードできる。

## 技術的な仕組み（開発・保守メモ）

### 全体構成
- **Firebase Realtime Database**（プロジェクト`rust-5bfa4`）：チーム・メンバー・ランキングデータをルームコード単位でリアルタイム共有
- **BattleMetrics公開API**（`api.battlemetrics.com`）：オンライン状況・プレイ時間ランキング。認証不要、ブラウザから直接呼べる（CORS許可済み）
- **rustoria.co内部API**（`api.rustoria.co`）：資源ファーム量・ワイプ履歴。CORSが許可されていないため、`rustoria-proxy-worker/`（Cloudflare Worker）を中継サーバーとして経由

### Cloudflare Workerプロキシ（`rustoria-proxy-worker/`）
`rustoria-resources-proxy.aulait11-17.workers.dev`にデプロイ済み。固定ルートテーブルのみ（`/resources`→資源リーダーボード、`/pvp`→キル・デス等、`/wipes`→ワイプ履歴、`/servers`→サーバー一覧）で、`server`クエリパラメータは既知のRustoria server id（26件）のホワイトリストと照合してから使うため、任意のURLに転送できるオープンリレーにはなっていない。CORSは`https://aulait1117-ux.github.io`のみ許可。Rustoriaがサーバーを追加・削除した場合は`worker.js`の`ALLOWED_SERVERS`を`/servers`エンドポイントの内容と再同期する必要がある。

修正・再デプロイ手順：
```
cd rustoria-proxy-worker
npx wrangler deploy
```
（Cloudflareアカウントへのログインが必要。`npx wrangler login`）

### Firebaseセキュリティルール
`.read`/`.write`はルート`false`、`/rooms/$roomCode`のみ`true`。ルームコードを知らないと読み書きできない（コード自体を推測されにくい6文字のランダム文字列にすることで安全性を担保する設計）。ルール変更はFirebaseコンソール（`console.firebase.google.com/project/rust-5bfa4/database/rust-5bfa4-default-rtdb/rules`）から行う。

### 既知の制約
- **rustoria.co利用規約の本文は未確認。** リサーチ部の調査・顧問室2名のレビュー（判断分かれ）を経て、社長判断で自動取得機能は継続稼働中（2026-07-06時点）。サーバー切り替え・キル/デス自動取得の追加でrustoria.co側への呼び出し種類が増えている点は、この既知リスクの範囲内という位置づけ。詳細は`08_品質管理部/検証ログ.md`・`11_顧問室/活動ログ.md`参照
- ロック画面のパスワードは簡易的な目隠しであり、本当のアクセス制御ではない
- GitHub Pagesのビルドが稀に`building`のまま止まることがある。反映されない場合は空コミット（`git commit --allow-empty`）でpushし直すと再トリガーできる。ビルドAPIのステータス表示が実際の反映より遅れることがあるため、反映確認は実際のページ内容（`curl`等）で行うこと

### 動作確認の方法
Puppeteer（`puppeteer-core` + ローカルのChrome実行ファイルを指定）で実ブラウザを操作させて確認するのが確実。CORSの制約上、rustoria.co連携部分の動作確認は本番URL（`https://aulait1117-ux.github.io/...`）に対して行う必要がある（localhostからはWorkerのCORS許可が通らない）。本番の共有ルームでテストした場合は、確認後に必ずテストデータを削除すること。
